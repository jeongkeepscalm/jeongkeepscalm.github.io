## 1강. 운영체제 소개  
  
* 운영체제
  - 대표적인 시스템 소프트웨어  
  - 컴퓨터 시스템의 자원을 관리하고 프로그램 동작을 위한 서비스를 제공하는 프로그램들의 모음  
* 운영체제 역할  
  1. 컴퓨터 시스템의 자원 관리(**컴퓨터 시스템을 효율적으로 운영하는 목적**)   
    - 저장장치에서 데이터 읽어오기  
    - CPU와 메모리를 효율적으로 관리  
  2. 사용자 지원  
    - 사용자가 내린 명령을 해석 후 실행  
    - 편의성 제공  
  
* 운영체제가 없던 초기의 컴퓨터 시스템  
  - 응용프로그램이 하드웨어를 직접 제어했었다.  
  - 응용프로그램 개발자는 하드웨어 제어 방법을 잘 알아야 했다.  
  - 여러 사용자가 하드웨어를 공유하는 경우 자원 분할이 어려웠다.  
* 운영체제가 있는 컴퓨터 시스템  
  - **하드웨어와 응용프로그램의 매개체**  
  - 운영체제가 컴퓨터 시스템의 자원 제어  
  
* 응용프로그램이 직접적으로 하드웨어 조작을 막기위해 커널모드와 사용자모드가 등장했다.  

* 커널모드(슈퍼바이저 모드) : 하드웨어를 직접 제어할 수 있는 CPU 명령어를 사용할 수 있는 모드. 운영체제의 커널이 동작하는 모드.  
* 사용자모드(보호 모드) : CPU 명령어를 사용할 수 없는 모드. 응용프로그램이 동작하는 모드.  
  
* 커널(kernel) : 커널 모드에서 동작하는 운영체제의 핵심 요소. 응용프로그램과 하드웨어 수준의 처리사이의 가교역할. '커널이 운영체제다' 라고 생각할 수 있지만 구성 방식에 따라 차이가 있다.   
* 일체형커널 ( 커널 == 운영체제 )  
	운영체제의 모든 서비스가 커널 내 포함.   
	커널 내부 요소들의 효율적으로 상호작용.  
	한 요소라도 오류가 발생하면 시스템 전체에서 장애 발생 가능.  
	( UNIX, Linux.. )  
* 마이크로커널   
	운영체제 요소의 대부분을 커널 외부로 분리.  
	커널 안에는 가장 핵심적인 요소들만 있다. ( 메모리 관리, 멀티태스킹, 프로세스 간 통신(IPC) 등.. )  
	새로운 서비스를 추가하여 운영체제를 확장하기 쉽다.  
	유지보수가 용이하여 안정성이 우수하다.   
	커널 외부 요소들 사이는 IPC가 필요하여 성능저하가 발생한다. (외부에서 내부의 IPC 를 통해서 가야하므로)  
  
* 시스템 호출   
	응용프로그램이 하드웨어에 대한 제어가 필요할 때, 이용한다.  
	운영체제에 서비스를 요청하는 메커니즘.  

### 운영체제 구성
  
* 프로세스 관리자 ( 실행 중인 프로그램을 관리한다. )  
	프로세스의 생성 및 삭제  
	CPU 할당을 위한 스케줄 결정  
	프로세스의 상태를 관리하며 상태 전이 처리  
	메모리에 올라와있는 프로그램의 순서를 스케줄링이 결정한다.   
* 메모리 관리자  
	차지하고 있는 메모리와 겹치지 않게 메모리 공간을 부여해서 할당 및 회수한다.  
	메모리 공간을 보호한다.   
* 장치 관리자   
	컴퓨터 시스템의 모든 장치에 할당, 작동, 반환해준다. ( 하드 디스크, SSD, 키보드, 마우스, 프린터 등.. )  
* 파일 관리자  
	저장장치의 공간 관리  
	파일의 접근 제한 관리  
  
### 운영체제의 유형
  
* 일괄처리 운영체제 (batch processing)  
	작업을 모아서 순서대로 처리하는 방식  
	전체적인 작업 처리 속도가 향상됨  
	나중에 들어온 중요도 높은 작업은 우선순위에 밀려 기다리게 된다.  
* 시분할 운영체제 (Time-sharing) == 대화형 운영체제 (interactive)  
	각 프로그램을 한 번에 조금씩 수행하는 방식  
	응답시간이 일괄처리 운영체제보다 크게 단축된다.  
* 실시간 운영체제 (real-time) (RTOS)
  원하는 시간 내에 프로그램 결과를 얻을 수 있는 방식. 시분할 운영체제는 응답시간이 빨라졌지만 실시간 운영체제는 최종적인 결과물을 받는 시간이 빨라졌다.  
  처리결과가 현재의 결정에 영향을 주는 환경에서 사용된다. ( 미사일 제어 시스템, 증권거래 관리 시스템 등.. )  
  우선순위가 높은 작업을 최우선처리할 수 있는 기법.
* 분산 운영체제  
  분산 시스템을 관리하기 위한 운영체제.
  분산 시스템 : 2개 이상의 컴퓨터 시스템이 네트워크로 서로 연결되어 서로의 자원을 이용하는 시스템  
  다른 컴퓨터 시스템의 자원을 이용하는 것이 마치 자신 컴퓨터 시스템에 있는 자원을 이용하는 것처럼 가능하게 한다.

## 2강. 프로세스와 쓰레드  
  
### 프로세스 개요    
  
  Hwp.exe 처럼 저장장치에 있을 때는, 프로그램에 관련된 명령어 목록이 들어있다고 보면 된다. (용량이 매우 작음)  
  Hwp.exe 가 실제 프로세스가 되면, 그 프로그램 자체에 부가적인 다양한 데이터들이 더 필요해서 메모리 용량이 훨씬 커지게 된다.  
  
* 프로세스
  실행중인 프로그램  
  동작을 하는 능동적 개체  
  운영체제로부터 자원을 할당 받아 동작한다.  
* 프로그램 : 동작하지 않는 정적, 수동적 개체  
=> 정적이였던 프로그램이 메모리 상에 올라와서 여러가지 자원을 할당 받고, 최종적으로 CPU도 할당받아서 실제 명령을 실행하는 동작을 한다.  
  
* 운영체제(프로세스 관리자)가 처리하는 작업
  프로세스를 생성 및 종료  
  프로세스를 실행시키기 위한 스케줄링 작업  
  프로세스의 상태 관리  
  
* 메모리 구조  
  코드 영역  
    프로그램 자체  
  데이터 영역  
    프로그램 실행 시 필요한 데이터를 보관한다.( 상수나 변수의 값, 서브프로그램 호출 상태 등.. )  
      정적 데이터 영역  
      스택 영역  
      힙 영역 
  
* 프로세스 제어 블록(PCB : Process Control Block) : 운영체제가 프로세스를 관리하기 필요한 정보를 가지고 있다.  
  프로세스번호(PID)  
  프로세스 상태  
  프로그램 카운터(PC) : 다음에 어떤 처리를 해야 하는지 알 수 있다. 제어의 흐름을 볼 수 있다.    
  레지스터 : 여러 프로세스가 번갈아 실행되는 경우 레지스터에 저장되어있던 값을 활용한다.  
  메모리 관리 정보  
  프로세스 우선순위  
  회계 정보 ...  
  
* 프로세스 상태  
  <img src="/assets/img/processStatus.png" width="800px" />
* 디스패치 : 특정 프로세스에 cpu를 할당해주는 과정  
  
* 프로세스 생성 방법  
  1. 사용자가 직접 프로그램 실행  
  2. 한 프로세스가 다른 프로세스 생성(부모프로세스가 자식프로세스 생성) : 프로세스 생성 시스템 호출 이용  
  
  * UNIX, Linux 일 경우  
  UNIX, Linux는 fork() 라는 시스템호출을 사용한다.  
    fork() : 부모프로세스를 복제해서 PID만 다른 자식프로세스를 생성한다. 자식프로세스의 PID를 return 한다.  
    fork()된 자식프로세스에 다른 일을 시키고 싶으면, fork() 다음에 exec()라는 시스템호출 하면 된다.  

  ```java
  // 메모리구조 내 코드영역
  // 만약 자식프로세스라면 program2라는 프로그램을 실행시켜라. 
  pid = fork();
  if (pid == 0) 
    exec(program2);
  ```

  * Windows 일 경우  
    CreateProcess()  
      자식프로세스는 처음부터 새로운 프로그램으로 생성
    
  ```java
  // 메모리구조 내 코드영역
  // 해당 자식프로세스는 곧바로 program2를 실행시켜라.
  CreateProcess(program2, ... ,&pid);
  ```
  
* 프로세스 종료 방법  
  프로세스가 모든 처리를 완료할 때( 정상적 종료 )  
  부모프로세스가 자식프로세스를 강제 종료할 수 있다.    
    자식의 PID를 이용해서 프로세스 종료 시스템 호출한다.  
  부모 프로세스가 종료되는 경우 운영체제가 자식 프로세스를 종료할 수 있다.

### 쓰레드  
  
* 쓰레드(Thread)
  프로세스 내에 다중처리를 위해 제안된 개념  
  디스패칭의 단위  
  하나의 프로세스에는 하나 이상의 쓰레드가 존재한다.  
  전통적인 프로세스와는 다르게 쓰레드가 PC를 가지고 있다.  
  실행에 필요한 최소한의 정보만 가진다.  
    PC를 포함한 레지스터 값  
    상태정보  
    스택 영역  
  나머지 정보는 프로세스에 두고 다른 쓰레드와 공유  
<img src="/assets/img/thread.png" width="500px" />
  
* 다중 쓰레드로 구성된 프로세스  
  멀티 CPU 또는 멀티코어 컴퓨터 시스템
    코어 3개 == CPU 3개. 즉, 3개의 쓰레드가 동시에 처리 가능.

## 3강. 프로세스 스케줄링
  
* 스케줄링 : 여러가지 작업의 처리순서를 결정한다.  
  프로세스 스케줄링  
  디스크 스케줄링  

* 프로세스 스케줄링 : 여러 프로세스가 주어졌을 때, 프로세스 처리 순서를 결정하는 것이다.  
  
<img src="/assets/img/stageOfScheduling.png" width="600px" />
  
디스패치는 준비큐에 있는 프로세스를 CPU에 할당하는 작업이므로 하위단계 스케줄링의 주체는 디스패처이다.  
  
### 스케줄리의 목표
  
* 스케줄링의 기본 목표  
  공정성 : 모든 프로세스가 적정 수준에서 CPU 작업을 할 수 있게 함.  
  균형 : 시스템 자원이 충분히 활용될 수 있게 함.  
  
* 운영체제 유형에 따른 스케줄링의 목표  
  일괄처리 운영체제 : 처리량 극대화, 반환시간(프로세스 생성~종료) 최소화, CPU활용 극대화.  
  시분할 운영체제 : 빠른 응답시간, 과다한 대기시간 방지.  
  실시간 운영체제 : 처리기한 맞춤.  
  
* 스케줄링 목표에 따라 우선적으로 고려해야할 기본적인 정책  
  
  * 선점 스케줄링 정책(preemptive)   
    실행 중인 프로세스에 interupt를 걸고 다른 프로세스에 CPU를 할당할 수 있는 스케줄링.  
    높은 우선순위의 프로세스를 바로 처리할 수 있다.(실시간 시스템, 시분할 시스템)  
    문맥교환에 따른 오버헤드가 발생한다. 그러므로 운영체제는 문맥교환이 매우 빠르게 실행되도록 만들어져야 한다.   
      문맥(context) : CPU의 모든 레지스터와 기타 운영체제에 따라 요구되는 프로세스 상태  
      문맥교환(context switching) : 현재 프로세스의 문맥을 PCB에 저장하고 다른 프로세스의 PCB를 읽어 문맥을 복원하는 작업이다.  
      오버헤드 : 추가적으로 시간,메모리,자원이 사용되는 현상.  
  
  * 비선점 스케줄링 정책(nonpreemptive)  
    실행 중인 프로세스를 바로 준비 상태로 문맥교환을 할 수 없다. ( 실행 -> 종료 / 실행 -> 대기 (o) )
    강제적인 문맥교환이 없기에 오버헤드가 없다.  
  
### 스케줄리의 평가 기준  
  
* 평균대기시간 : 프로세스가 준비큐에 들어가 있는 시간의 합의 평균값.  
* 평균반환시간 : 프로새스 생성에서 종료까지 걸리는 시간의 합의 평균값.  
  
### 스케줄링 알고리즘  
  
* FCFS(First-Come First-Served)  
  비선점 방식 (시분할 운영체제나 실시간 운영체제에 부적합)   
  준비 큐에 도착한 순서에 따라 디스패치.  
  프로세스의 도착 순서에 따라 반환시간이 크게 변한다.  
  <img src="/assets/img/fcfs.png" width="600px" />
  
* SJF(Shortest Job First)  
  비선점 방식 (시분할 운영체제나 실시간 운영체제에 부적합)   
  준비 큐 내 프로세스 중 가장 짧은 실행시간으로 예상되는 프로세스를 먼저 디스패치한다.  
  일괄처리 환경에서 구현하기 쉽다.  
  먼저 처리할 CPU 시간을 예상하기 어렵다.
  <img src="/assets/img/sjf.png" width="600px" />
  
* SRT(Shortest Remaining Time)  
  선점방식 ( 문맥교환으로 SJF보다 오버헤드가 크다. )  
  SJF보다 평균대기시간이나 평균반환시간에 효율적이다.  
  CPU 시간을 예상하기 어렵다.  
  <img src="/assets/img/srt.jpg" width="600px" />
  
* RR(Round Robin)  
  선점방식  
  FCFS처럼 먼저 도착한 순서대로 디스패치 하지만, 정해진 시간할당량이 주어지고 시간할당량 내 종료하지 못한 프로세스는 준비큐의 마지막에 배치된다.  
  CPU를 독점하지 않고 공평하게 이용한다.(시분할 운영체제에 적합)  
  시간 할당량이 너무 크면, FCFS와 다를 바 없다.  
  시간 할당량이 너무 작으면, 잦은 문맥교환 발생으로 오버헤드가 커진다.  
  <img src="/assets/img/rr.jpg" width="600px" />
  
* HRN(Highest Response Ratio Next)  
  비선점방식  
  준비큐에서 기다리는 프로세스 중 응답비율이 가장 큰 것을 먼저 디스패치한다.  
  응답비율 = 대기시간/예상실행시간 + 1;  
  => 준비큐에 오레 기다린 프로세스이거나 CPU 사이클이 짧을 수록 응답비율이 크다.  
  HRN = SJF의 기능 + 오래 기다린 프로세스가 먼저 처리될 수 있게하는 기능.  
  CPU 시간을 예상하기 어렵다.  
  <img src="/assets/img/hrn.jpg" width="600px" />

* 다단계 피드백 큐 스케줄링    
  선점 방식  
  I/O 중심 프로세스와 연산 중심 프로세스의 특성에 따라 **서로 다른 시간 할당량 부여**  
  1 ~ n 단계까지 준비큐 존재  
  1단계 준비큐에 프로세스가 다 실행되고 종료되지 못한 프로세스들은 2단계 준비큐로 넘어가게 되고 ... n단계 준비큐까지 이어진다.  
  단계가 커질수록 시간 할당량도 커진다.  
  I/O 중심 프로세스에 높은 우선권이 유지된다.  
  연산 중심 프로세스는 낮은 우선권을 갖지만 긴 시간 할당량을 갖는다.

## 4강. 병행 프로세스 1
  
### 병행 프로세스의 개요  
  
* 병행성 : 여러 프로세스/쓰레드가 동시에 수행되는 시스템 특성이다.  
* 병행 프로세스 : 여러 프로세스/쓰레드가 동시에 수행.  
  CPU가 1개 일 경우(인터리빙 형식)  
    여러 프로세스가 실행될 수는 있지만 CPU에서의 처리는 한 프로세스만 처리하고 있다.(RR스케줄링과 비슷하게 처리)  
  CPU가 여러개 일 경우(병렬처리 형식)  
    CPU의 개수만큼 프로세스 동시에 처리 가능하다.  
  즉, 하나의 CPU는 하나의 프로세스만 처리할 수 있다.  
  
* 멀티프로세서(== 여러 CPU일 경우) 시스템에서 메모리구조에 따라 나뉜다.  
  * 강결합 시스템
    여러 CPU들이 **하나의 메모리**를 공유한다.(공유 메모리 구조)  
  * 약결합 시스템
    CPU 마다 메모리가 별도로 존재한다.(분산 메모리 구조)  
    CPU끼리 정보를 주고 받을 때, 통신선을 통한다.  
  
* 독립 프로세스  
  각각의 프로세스가 자신의 일만 한다. 다른 프로세스에 영향을 주고 받지 않는다.  
  데이터 및 상태를 다른 프로세스들과 공유하지 않는다.  
  결정적. 실행 결과는 입력에 의해서 결정된다.  
  재생가능. 같은 입력에 대하여 항상 같은 실행결과를 리턴한다.  
* 협력 프로세스  
  수행중인 다른 프로세스와 영향을 주고 받는다.  
  데이터 및 상태를 다른프로세스들과 공유한다.  
  비결정적. 실행결과는 실행순서에 좌우된다.  
  재생불가능. 같은 입력에 대하여 항상 같은 실행결과를 보장할 수 없다.   
  
### 병행성 문제  
  
협력 프로세스일 경우(생기는 문제)  
1. 상호배제 해야한다.  
  2개 이상의 프로세스가 동시에 임계영역을 수행하지 못하도록 해야 한다.  
  * 임계영역 : 두 개 이상의 프로세스가 동시에 공유 자원에 접근을 허용하는 프로그램 코드 영역.  
  ex ) A코드와 B코드를 동시에 실행할 경우 공유자원의 값이 실행순서에 따라 달라질 수 있다. 그렇기 때문에 A코드 임계영역에 있을경우, B코드는 실행되선 안된다.  
  즉, 임계영역의 부분이 상호 배제가 될 수 있도록 잘 처리를 해주어야 한다.  
2. 동기화 해야한다.   
  2개 이상의 프로세스들의 처리 순서를 결정하는 것이다. 
  ex ) 입금시, 출금프로세스 다음에 입금프로세스를 태워야하는데, 그 반대가 될 경우 문제가 생긴다.  
3. 통신을 해야한다.    
  프로세스들이 데이터를 공유하기위해 반드시 필요하다.  
  
### 세마포어  
  
* 세마포어( == 정수형 공용변수)  
  상호배제와 동기화 문제를 해결할 수 있는 도구이다.  
  Dijkstra가 제안.  
  임계변수에 들어갈 수 있는 여부에 따른 정수값 혹은 사용 가능한 자원의 수를 공용변수로 저장할 수 있다.  
  상황에 맞춰 0 이상인 정수로 초기화한다.  
  두 기본연산 p와 v에 의해서만 사용된다. 기본연산은 중간에 중단되지 않고 하나의 단위로 처리된 후에 다음 작업을 한다.  
    
  * 연산 p  
    검사 또는 감소시키려는 시도이다.  
    ```java
    void p(semaphore s) {
      if (s > 0) {
        s--;
      } else {
        현재 프로세스를 대기시킨다;
      }
    }
    ```
  
  * 연산 v  
    증가.
    ```java
    void v(semaphore s) {
      if (대기 중인 프로세스가 없음) {
        s++;
      } else {
        대기 중인 프로세스 1개를 진행한다;
      }
    }
    ```
  
  즉, 세마포어 마다 대기 큐가 필요하다.  
  
* 상호배제 해결 
  임계영역 수행 중에는, 한 프로세스만 실행되어야 한다.  
  1. 임계영역에서 실행 중인 프로세스가 끝날 시, p() 가 실행되어 --mutex 된다.  
  2. 임계영역에 접근하지 못한 프로세스들은 대기 큐에 순서대로 쌓여 FIFO 순서로 실행된다.  
  3. 대기 큐의 마지막 프로세스가 종료될 경우, v() 가 실행되어 ++mutex 해준다.

### 생산자-소비자 문제  
   
* 두 협력 프로세스 사이에 버퍼를 두고 생산자와 소비자의 상황을 다루는 문제  
  생산자 : 데이터를 버퍼에 넣음.  
  소비자 : 데이터를 버퍼에서 꺼냄.  
  버퍼 : 임시 저장 공간.  
  
* 버퍼에 여러 프로세스가 동시에 접근할 수 없다.  
  데이터를 버퍼에 넣을 동안 버퍼에서 데이터를 꺼낼 수 없다.  
  버퍼에서 데이터를 꺼낼 동안 데이터를 버퍼에 넣을 수 없다.  
  => **상호배제 필요**
* 유한 버퍼 문제  
  버퍼의 크기가 유한하다.  
  버퍼가 가득 찬 경우, 생산자는 대기해야한다.  
  버퍼가 비어 있을 경우, 소비자는 대기해야한다.  
  => **동기화 필요**

### 판독기-기록기 문제  
  
* 여러 협력 프로세스 사이에 공유자원을 두고 판독기와 기록기의 상황을 다루는 문제  
  판독기 : 데이터를 읽는 프로세스.  
  기록기 : 데이터를 쓰는 프로세스.(생산자, 소비자)   
  
* 하나의 기록기가 공유자원에 데이터를 쓰는 중에는, 다른 기록기/판독기는 공유자원에 접근 할 수 없다.  
  공유자원에 데이터를 쓰고 있을 동안에는 누구도 공유자원에 접근할 수 없다.  
  공유자원의 데이터를 읽고 있을 동안에는 누구도 공유자원에 데이터를 쓸 수 없다.  
  
* 여러 판독기는 동시에 공유자원에서 데이터를 읽을 수 있다.  
  단, a판독기가 공유자원의 데이터를 읽을 동안 x기록기가 대기중이고 b판독기가 들어 왔을 경우에는 정책에 따라 순서가 달라질 수 있다.  
    1. a판독기 -> b판독기 -> x기록기(**제 1판독기**)  
    2. a판독기 -> x기록기 -> b판독기(**제 2판독기**)    

  * 제 1판독기  
    우선순위가 판독기에 있음.  
    기록기의 기아상태 유발 가능.  
    기아상태 : 실행 될 가능성은 있지만 기한 없이 기다려야 되는 상태.  
    <img src="/assets/img/reader.jpg" width="600px" />
  
  * 제 2판독기  
    판독기 실행 중 기록기가 대기 중이고 다른 판독기가 들어왔을 경우, 대기 중인 기록기에 우선순위를 준다.  
    판독기가 기아상태 유발 가능.

### 프로세스 간 통신 (IPC)   
  
* InterProcess Communication  
* 병행 프로세스가 데이터를 서로 공유하는 방법  
  하나의 운영체제에서 두 방법을 함께 사용 가능하다.  
  1. 공유 메모리 방법  
    협력 프로세스가 공유 메모리 공간을 활용하여 동일한 변수를 사용한다.  
    대량 데이터 교환이 편하다. (고속 통신 가능)  
    공유 메모리를 공유했을 시 발생하는 문제점을 프로그래머들이 다 해결을 해주어야 한다.  
  2. 메시지 전달 방법  
    시스템호출 send(), receive() 사용.  
    send() 함수를 통해 커널로 전달 후 커널이 receive() 를 이용해 전달한다.  
    소량 데이터 교환에 적합하다.  
    메시지 전달로 인해 발생한 문제는 운영체제가 해결한다.  
      * 통신 링크 : 메시지가 지나다니는 통로  
      <img src="/assets/img/linkVolume.jpg" width="600px" />
        * 직접통신  
        두 프로세스가 직접 서로를 지정하여 메시지를 전달한다.  
        통신 링크가 하나만 자동으로 설정된다.  
        양방향의 통신링크를 가진다.  
          대칭형 주소 지정 : 프로세스명을 직접 지정해서 통신한다.  
          비대칭형 주소 지정 : 변수를 매개변수로 사용해서 통신한다.  
        * 간접통신  
        두 프로세스 사이에 둔 우편함을 통해 메시지를 전달한다.
